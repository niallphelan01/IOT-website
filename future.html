Future developments

Future developments in Intruder Alarm could be in the following areas

Programming of channels

Rather than hardcoding in devices into particular tasks, we could default the system to have a number of channels. These channels could be configured to be a door sensor, tamper or PIR via a json file called by the main program. As part of the json file then interface could be selected and the GPIO pins. Also an option for selecting live or unused would be useful. This in the end would allow for the adjustment of just one json file inorder to setup this system for other setups much more like mainstream systems.

Web interface

A web interface for setting the alarm, getting back information would be a great solution for such an application. This is something that would be a nice edition.

Additional devices

A heater could be added to the setup for the shed in order to keep the temperature from freezing on more sensitive equipment

Network scan

I have implemented a very simple network scan (arp-scan) in this application to see what devices are at home when the intruder alarm is activated. This could be expanded further maybe to see if certain phones are home then to ping them with an alert (email, text ,etc.)

Clearing of old photos and videos - Service dashboard

It will be important to have some web interface in order to delete S3 stored data. A web application could request deletion in the node.js application from the s3 bucket.

securing S3 bucket

For now the S3 bucket is public, this is only temporary until a method for securing this is achieved.

database

Move the database to a dedicated database server. Probably AWS EC2 instance with node.js and mysql installed on same.
